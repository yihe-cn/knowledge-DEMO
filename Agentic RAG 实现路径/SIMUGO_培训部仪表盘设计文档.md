## 1. 业务定位

### 1.1 仪表盘要解决的根本问题

SIMUGO 之前的产品形态里,客户登录后看到的是**练习数据**——学员练了几次、平均得分多少、谁练得多。

这是"练习工具"的视角,不是"能力平台"的视角。

新仪表盘要让客户登录后看到的是**自己企业的能力图谱**——
- 我们企业需要哪些能力(能力域 + KP)
- 这些能力的"教材"覆盖如何(KB 文档对哪些 KP 有标准定义)
- 员工掌握得怎么样(逐 KP 的练习表现分布)
- 哪些能力是硬骨头(反复栽倒、平均分低、SOP 缺失)
- 培训投入产生了什么结果(随时间的进步曲线)

**视角的根本变化**:从"看练习记录"变成"看能力建设进展"。这件事看似只是 UI,实际上是商业模式的转折点。

### 1.2 MVP 目标用户:培训负责人

明确聚焦,不试图同时服务多个角色:

| 角色 | 关心 | MVP 服务? |
|------|-----|----------|
| 培训负责人 / 培训部 VP | 员工成长、培训效果、续约说服 | **核心** |
| 合规官 | 合规边界、数据审计 | 由独立的合规审计后台服务 |
| 业务部门主管 | 自己团队的短板、业务绩效 | v2 |

聚焦培训负责人的理由:这是 SIMUGO 续约决策的关键人,平台叙事在他这里立住,整个产品就立住。

### 1.3 培训负责人的核心使用场景

文档下面的设计都围绕这五个场景:

**场景 A:日常巡查**(每周 1-2 次,~5 分钟)
"我们企业的整体能力建设进展怎么样?有没有什么异常需要我关注?"

**场景 B:深度分析**(每月 1-2 次,~30 分钟)
"我要为下个月的培训计划做决策,哪些能力短板需要重点投入?"

**场景 C:向上汇报**(每季度 1 次,~2 小时准备时间)
"我要给总裁/董事会汇报本季度培训成果,需要一份高质量的能力建设报告。"

**场景 D:个体辅导**(不定期,~10 分钟)
"某个具体员工的表现需要 review,我要看他的能力剖面。"

**场景 E:文档管理**(不定期,~20 分钟)
"我要把新的合规手册上传到 KB,或者更新某条 SOP。"

### 1.4 不在 MVP 范围内的内容

- 业务主管视角的部门下钻
- 实时数据(MVP 用 T+1)
- 多个仪表盘可配置(每个客户都是同一个仪表盘)
- 钻取分析、自定义查询、临时 SQL
- 个性化首页布局
- 移动端响应式(MVP 仅桌面端)
- 数据导出为非 PDF 格式(Excel/CSV 留 v2)

## 2. 仪表盘整体结构

### 2.1 顶层导航(左侧栏)

```
┌─────────────────┐
│ [企业 LOGO]      │
│                  │
│ 📊 能力图谱       │  ← 首页,场景 A
│ 🎯 能力短板分析   │  ← 场景 B
│ 👥 员工能力剖面   │  ← 场景 D
│ 📄 知识库管理     │  ← 场景 E
│ 📑 报告中心       │  ← 场景 C
│                  │
│ ────────────     │
│ ⚙️ 平台配置       │  ← 配置项(钩子开关、文案 override)
│ 🔐 合规审计       │  ← 跳转合规审计后台
│                  │
│ [培训部 王经理]   │
└─────────────────┘
```

**设计判断**:
- "能力图谱"是首页,登录直接进——这是平台叙事的"门面"
- 不放在导航顶部的"练习数据"——MVP 阶段刻意降低练习相关数据的层级,扭转"练习工具"印象
- 5 个主功能 + 2 个辅助,数量是培训部用户能记住的极限

### 2.2 五大主页面的关系

```
能力图谱(首页)
    │
    ├─ 点击某能力域 ──→ 能力域详情(域内 KP 列表)
    │                        │
    │                        └─ 点击某 KP ──→ KP 详情(穿透到所有相关数据)
    │
    ├─ 异常标记 ──→ 能力短板分析(自动跳转到该短板)
    │
    └─ 点击 KPI ──→ 对应的明细页面


能力短板分析
    │
    ├─ 短板 KP 列表
    └─ 点击某 KP ──→ KP 详情(穿透)


员工能力剖面
    │
    └─ 员工列表 + 个体能力剖面


知识库管理
    │
    ├─ 文档列表
    └─ 文档详情(含 KP 绑定状态)


报告中心
    │
    └─ 报告列表 + 生成/查看/下载
```

**关键原则**:KP 是仪表盘里的一等公民,所有路径都能穿透到"KP 详情"页面。这是 KP 架构在客户侧的具象化。

## 3. 五大页面 wireframe 和设计

### 3.1 能力图谱(首页)

**目标**:让培训负责人在 30 秒内对企业能力建设有全局直觉。

**Wireframe**:

```
┌──────────────────────────────────────────────────────────────────┐
│ 能力图谱                                              [选择岗位 ▾] │
│                                                       医药代表 ▾    │
├──────────────────────────────────────────────────────────────────┤
│                                                                    │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │
│  │ KP 覆盖率    │ │ 学员活跃度   │ │ 平均掌握度   │ │ 硬骨头 KP    │ │
│  │             │ │             │ │             │ │             │ │
│  │   72%       │ │  85 / 120   │ │   68%       │ │   8 个       │ │

│                                                                    │
├──────────────────────────────────────────────────────────────────┤
│ 能力域分布                                                          │
│                                                                    │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │                                                              │  │
│  │   [能力域饼图]                                                │  │
│  │   外圈:KP 数量分布                                           │  │
│  │   内圈:掌握度颜色(深绿=高、浅绿=中、灰=低、红=异常)          │  │
│  │                                                              │  │
│  │   点击任一能力域 → 展开右侧详情面板                            │  │
│  │                                                              │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                    │
├──────────────────────────────────────────────────────────────────┤
│ 需要关注                                                            │
│                                                                    │
│  ⚠ 价格异议处理 · 价值锚定话术                                       │
│     12 名学员近期失败,平均得分 54 分,SOP 文档缺失                   │
│                          [查看详情 →]  [上传 SOP →]                 │
│                                                                    │
│  ⚠ 不良反应应对 · 24 小时上报流程                                    │

│                          [创建场景 →]                              │
│                                                                    │
└──────────────────────────────────────────────────────────────────┘
```

**核心 UI 元素**:

**顶部 KPI 卡片(4 个)**

| KPI | 计算 | 价值 |
|-----|-----|------|
| KP 覆盖率 | 至少有一个 source 绑定的 KP / 总 KP 数 | 反映企业能力图谱的完整度 |
| 学员活跃度 | 最近 30 天有练习的学员数 / 总学员数 | 反映学员是否在使用平台 |
| 平均掌握度 | 所有学员所有 KP 的平均 mastery_estimate | 反映整体能力建设效果 |
| 硬骨头 KP | mastery 中位数 < 阈值 且 encounter ≥ 5 的 KP 数 | 反映需要重点投入的方向 |

每个 KPI 卡片包含:
- 主数字(大字号、视觉焦点)
- 较上月/上周的变化(箭头 + 变化值)
- 异常标记(红/黄/绿,基于阈值)
- 点击 → 跳转到对应的明细页

**中部能力域饼图**

设计要点:
- **不要用普通饼图**——普通饼图只表达"数量分布",信息密度太低
- **用双层环形图**——外环表达 KP 数量、内环表达掌握度,**单图传递两层信息**
- 颜色编码:深绿(掌握度 > 80%)、浅绿(60-80%)、灰(40-60%)、红(< 40% 且样本足)
- 鼠标 hover 显示该能力域的具体数字
- 点击某能力域 → 右侧滑出详情面板(域内 KP 列表 + 各项指标),不切页

**底部"需要关注"模块**

这是首页最重要的产品创新——**不是被动展示数据,而是主动告诉培训负责人"你下一步该做什么"**。

每条关注事项的结构:
- 严重程度图标
- 一句话描述(具体的 KP + 问题 + 数据)
- 直接行动 CTA 按钮(查看详情 / 上传 SOP / 前往复核 / 创建场景)

关注事项的类型(MVP 阶段实现):

| 类型 | 触发条件 | 推荐行动 |
|------|---------|---------|
| KP 掌握度急剧下降 | mastery 月环比下降 > 10% | 查看 KP 详情、安排培训 |
| SOP 文档缺失 | KP 被反复栽倒但无 source 绑定 | 上传 SOP |
| 复核待办 | 文档已上传但有 chunk 待复核 | 前往复核 |
| chatbot 问得多但无场景 | 30 天提问 > 30 次但无场景绑定 | 创建场景 |
| 长期未练习的硬骨头 KP | 该 KP > 60 天无练习记录 | 加入下月培训计划 |

**关注事项排序逻辑**:严重程度 > 行动可立刻执行的优先 > 数据新鲜度。最多展示 5 条,多了下拉"查看更多"。

**右上角岗位筛选器**

第一版只有"医药代表"一个选项,但 UI 上要给筛选器,为后续岗位扩展占位。

**核心 UI 元素**:

**短板 KP 列表**

每条记录的"严重程度"由综合指标决定:

| 等级 | 触发条件 |
|------|---------|
| 🔴 高 | 掌握度 < 50% 且 encounter ≥ 5,或月环比下降 > 15% |
| 🟡 中 | 掌握度 50-65% 且 encounter ≥ 5,或月环比下降 5-15% |
| ⚪ 观察 | 掌握度 65-75%(还可以但需要关注) |
| 不展示 | 掌握度 > 75% 或 encounter < 5 |

"建议行动"列由数据综合判断,MVP 阶段简化逻辑:

| 状况 | 建议 |
|------|-----|

## 4. 关键功能详细设计

### 4.1 KPI 计算细节

**KP 覆盖率**:
- 分子:`SELECT COUNT(DISTINCT kp_id) FROM knowledge_point_source WHERE enterprise_id = ? AND confirmed = TRUE`
- 分母:`SELECT COUNT(*) FROM knowledge_point WHERE industry = ? AND visible_for_enterprise(?)`
- 显示:`{numerator} / {denominator} = X%`

**学员活跃度**:
- 分子:`SELECT COUNT(DISTINCT user_id) FROM practice_session WHERE enterprise_id = ? AND completed_at >= NOW() - INTERVAL 30 DAY`
- 分母:`SELECT COUNT(*) FROM users WHERE enterprise_id = ? AND status = 'active'`

**平均掌握度**:
- `SELECT AVG(mastery_estimate) FROM learner_kp_profile WHERE enterprise_id = ? AND encounter_count >= 1`
- 注意:只算"有过遭遇"的 KP,避免被未练习的 KP 拉低

**硬骨头 KP 数**:
- 条件:`mastery_estimate < 0.5 AND encounter_count_aggregate >= 5`(企业级聚合)
- 实现需要预聚合表,见 5.3

### 4.2 "需要关注"模块的检测逻辑

后台需要一个**关注事项检测器**(Celery 周期任务,每日凌晨运行):

```python
class AttentionItemDetector:
    """每日检测各企业的需要关注事项"""

    async def detect_for_enterprise(self, enterprise_id: UUID) -> list[AttentionItem]:
        items = []

        # 检测 1:KP 掌握度急剧下降
        items.extend(await self._detect_mastery_drop(enterprise_id))

        # 检测 2:SOP 缺失
        items.extend(await self._detect_sop_missing(enterprise_id))

        # 检测 3:复核待办
        items.extend(await self._detect_review_pending(enterprise_id))

        # 检测 4:chatbot 问得多但无场景
        items.extend(await self._detect_no_scenario_high_query(enterprise_id))

        # 检测 5:长期未练习的硬骨头
        items.extend(await self._detect_stale_hard_kp(enterprise_id))

        # 排序:严重程度 + 可立即行动
        items.sort(key=self._priority_key)

        return items[:10]  # 至多展示 10 条,UI 展示前 5 条
```

每条 AttentionItem 包含:
- type(检测类型)
- severity(高/中/低)
- title(一句话描述,模板渲染)
- action_url(行动 CTA 跳转 URL)
- action_label(按钮文案)
- triggered_data(详细数据,供前端展示和审计)

**结果物化到表**:

```sql
CREATE TABLE attention_item (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    enterprise_id VARCHAR(64) NOT NULL,
    type VARCHAR(50) NOT NULL,

    action_label VARCHAR(50),

    resolved_at TIMESTAMP,   -- 用户完成行动后自动 resolve
```

### 4.4 报告生成

**报告生成流程**:

```
用户选择报告类型和参数
    ↓
后端 Celery 任务启动
    ↓
查询所需数据(预聚合表 + 实时计算)
    ↓
组装报告数据结构
    ↓
渲染 HTML 模板
    ↓
HTML → PDF(weasyprint 或 wkhtmltopdf)
    ↓
PDF 上传到阿里 OSS
    ↓
通知用户(站内消息 + 邮件)
    ↓
用户下载/分享
```

**技术选型**:

- 报告引擎:**weasyprint**(Python,HTML/CSS → PDF,中文支持好)
- 备选:wkhtmltopdf(更老但稳定)
- 模板引擎:Jinja2
- 图表:服务端生成的 SVG(用 ECharts 服务端渲染 + Pyppeteer 转 PNG/SVG)

**季度能力建设报告的标准结构**(MVP):

1. **执行摘要**(1 页):本季度核心数字 + 三大亮点 + 三大短板
2. **整体进展**(1-2 页):KPI 同比环比 + 趋势图
3. **能力图谱概览**(1-2 页):能力域分布饼图 + 关键能力域分析
4. **TOP 强项 KP**(1 页):本季度表现最好的 5 个 KP + 数据支撑
5. **TOP 短板 KP**(1-2 页):需要重点投入的 5 个 KP + 建议行动
6. **资源建设进展**(1 页):KB 文档新增、KP 覆盖率提升、场景库扩充
7. **学员活跃度分析**(1 页):活跃学员数、新增学员、练习时长分布
8. **下季度建议**(1 页):基于数据的具体建议(由数据规则生成,不是 LLM 生成)
9. **附录:数据明细表**(2-3 页)

**为什么不用 LLM 生成报告内容**:

MVP 阶段坚持用**模板渲染 + 数据填充**,不用 LLM 生成文案。原因:

- 报告是要给客户高层看的,内容准确性至关重要
- 数据驱动的模板能保证 100% 准确
- LLM 生成可能加幻觉(尤其在涉及具体数字描述时)
- 客户合规官可能审核报告内容,模板版本可追溯
- 节省 LLM 成本

V2 可以考虑用 LLM 做"基于数据的洞察生成",但 MVP 不做。

**报告生成的工作量估计**:这是较大的工程模块,约 3-4 人周(数据查询 + 模板 + PDF 生成 + UI)。考虑是否纳入 MVP 见决策请求。

## 5. 数据查询和性能

### 5.1 预聚合策略

仪表盘数据**全部 T+1**,通过每日凌晨的 Celery 任务预聚合到物化表。

理由:
- 培训部用户不要分钟级实时
- 仪表盘页面打开要快(< 1 秒),实时计算多个聚合不可行
- 减轻线上数据库压力

### 5.3 预聚合任务设计

```python
class DashboardAggregationPipeline:
    """每日凌晨运行,聚合所有企业的仪表盘数据"""

    async def run_daily(self, snapshot_date: date):
        enterprises = await self._get_active_enterprises()

        for enterprise_id in enterprises:
            await self._aggregate_kp_level(enterprise_id, snapshot_date)
            await self._aggregate_domain_level(enterprise_id, snapshot_date)
            await self._aggregate_overview_level(enterprise_id, snapshot_date)
            await self._detect_attention_items(enterprise_id)
            await self._aggregate_peer_kp(enterprise_id)  # 钩子四数据底座
```

**Celery beat 配置**:
- 每日凌晨 2 点启动(数据时延约 2 小时,可接受)
- 单企业聚合失败不影响其他企业
- 失败重试 3 次后告警
- 全量聚合预计 < 30 分钟(10 客户规模)

## 6. 隐私和权限

### 6.1 数据可见性约束

延续 KB 开发文档和钩子文档的隐私设计:

| 数据 | 培训部可见性 |
|------|-----------|
| 学员练习表现(KP 维度) | ✓ 可见(汇总和个体) |
| 学员练习转录详情 | △ 仅当配置允许时可见 |
| 学员 chatbot 提问次数(汇总) | ✓ 可见 |
| 学员 chatbot 提问内容/KP 标签 | ✗ 不可见(MVP) |
| 学员个人偏好设置 | ✗ 不可见 |

具体在仪表盘的体现:
- 员工剖面页只显示提问"次数",不显示提问"内容"或具体的提问 KP 分布
- KP 详情页的"问答画像" Tab 是聚合数据,不下钻到学员
- 任何导出报告都不暴露上述禁止字段

### 6.2 权限角色

MVP 阶段的角色定义:

| 角色 | 仪表盘可见 | 可操作 |
|------|---------|------|
| 平台超级管理员(SIMUGO) | 全部企业 | 全部 |
| 客户管理员(IT) | 本企业 | 全部 |
| 培训负责人 | 本企业 | 查看 + 生成报告 + 上传文档 |
| 培训助理 | 本企业 | 查看 + 上传文档 |
| 合规官 | 本企业 | 查看 + 跳转到合规审计后台 |
| 业务部门主管 | 本企业 + 限本部门 | 查看(MVP 不开放,v2 启用) |

实现:基于现有 SIMUGO 认证体系扩展角色,每个 API 接口在路由层做权限校验。

### 6.3 跨企业数据隔离

强约束:任何查询都必须带 `enterprise_id` 作为过滤条件。

实现:
- ORM 层全局过滤器(SQLAlchemy 的 `session.query()` 自动 inject `enterprise_id`)
- 跨企业的聚合数据(钩子四的 platform 视图)走独立服务接口,有独立权限
- 渗透测试:模拟 A 企业用户尝试访问 B 企业数据,所有请求都应拒绝

## 8. 设计判断说明

本文档基于以下我做出的设计判断,产品负责人 review 时可以挑战:

| 判断 | 原因 |
|------|------|
| MVP 聚焦培训负责人一个角色 | 续约关键人,聚焦做透比平均分摊到三个角色好 |
| 能力图谱是首页,不是练习数据 | 平台叙事的"门面",刻意降低练习相关数据层级 |
| "需要关注"模块作为首页核心 | 主动告诉客户下一步做什么,差异化能力 |
| 双层环形图代替普通饼图 | 单图传递两层信息,信息密度高 |
| 报告 MVP 用模板渲染不用 LLM | 准确性要求高,LLM 风险大 |
| 所有数据 T+1 不实时 | 培训部场景下完全够用,大幅简化技术复杂度 |
| 不展示 chatbot 提问内容到培训部 | 心理体感保护一致性 |
| 业务主管视角 v2 | MVP 资源聚焦,主管视角是数据下钻问题,可加 |
