## 一、过程中观察到的系统问题

| # | 问题 | 表现 | 根因 |
|---|---|---|---|
| **P1** | **Celery worker 加载的是旧代码**，新版 `load_pdf()` 不生效 | 第一次走 `/api/admin/kb/upload` 直接失败，错误信息 `loader for .pdf not implemented (MVP only supports .pptx)`——这条字符串在当前 `loaders.py:81` 已被改为 `MVP supports .pptx and text pdf`。失败记录 `doc_id=9` 仍在 DB（也证明 `doc_id=8` 早就因同样原因失败） | worker 自 8:55 PM 启动后未重启，源代码改动未热加载 |
| **P2** | **chunk meta 中 `slide_indices` 全为空 `[]`** | 所有 42 个 chunk 的 meta 都是 `{'titles': [...], 'slide_indices': []}` | PDF 走的是 `load_pdf` 写入 `page_index`，但 chunker（`chunker.py`）合并 meta 时只读 `slide_index` 键，没读 `page_index`，导致页码定位信息全部丢失 — 后续 KP 无法回溯到具体页 |
| **P3** | **`title` 字段被页脚污染** | 全部 chunk 的 titles 几乎都是 `©  Copyright indevp®  逸禾® CONFIDENTIAL c`（每页第一行的版权页眉） | `_make_title` 取每页第一行非空文本，pypdf 顺序下版权行最先 |
| **P4** | **chunk 跨页粗暴合并** | chunk #2 把第 5/6/8 页（个人小组演练 / 体验码 / 班级报告）合并成一段 | 这些页文本极短（30–60 字），被 chunker 合并至 ≥120 token 阈值才停，章节边界被破坏 |
| **P5** | **图表/截图/流程图内容完全丢失** | 第 14–18 页 5 张「预设主题展示」截图、第 3 页 4 步流程图、第 35 页对比图等，pypdf 只能拿到标题，正文（卡片名/图例）全无 | `load_pdf` 不做 OCR，PPT 导出 PDF 中文字常作位图保存 |
| **P6** | **KP 抽取存在系统性遗漏与误粒度** | 详见第三节：8 个明显 KP 未被抽取；4 模式漏了"学习导师" | LLM 每 8 chunks 一批，无全局视角；prompt 偏「术语-定义」结构，对扁平的列举式内容（如"四种模式""三大挑战"）易抽部分丢部分 |

## 四、典型失败案例深挖（3 个）

### 案例 A：「四种 AI 角色模式」抽掉一个
- **原文**（p.28，chunk #20–21 一起）：访谈助手 / 学习教练 / 陪练伙伴 / 学习导师 ——四个并列模式各有一段说明。
- **系统结果**：#96 #97 #98 三个独立 KP，**第 4 个「学习导师」连同其特有的「看不见的观察员 AI」一并丢失**。
- **根因**：LLM 在单次 batch 内对扁平列举不做"枚举完整性"校验；prompt 没要求"如果文中出现 N 个并列项，必须输出 N 个 KP"。

### 案例 B：「SIMUGO® 算法」漏第 5 环节
- **原文**（p.37）：结构化与系统化 / 意图归类萃取 / 融入销售技巧模型 / 识别与提炼 / 表达内容重塑 / **低效表达纠偏**（共 5 步，最后一步另起一列）。
- **系统结果**：#104 定义里只提到「意图归类、融入销售模型、表达内容重塑和低效表达纠偏等步骤」——其实把"识别与提炼"漏了，且把"5 步"压缩到一个 KP 的定义里。
- **根因**：P3/P4 — chunk #33 把 p.37+p.38 合并，pypdf 把图的列顺序打乱，LLM 在乱序文本上漏读。

### 案例 C：dedup 误判 — 「客户洞察 vs 客户痛点」
- 同一个语义在系统中变成 #108「客户洞察与需求挖掘能力」 + #109「客户痛点挖掘技巧」两个 KP，定义重叠 70%+。
- **根因**：embedding 余弦阈值 0.85 对中文销售技巧词汇区分度不足，**`挖掘技巧` 与 `挖掘能力` 在 BGE-small 上余弦可能仅 0.82**，刚好低于阈值。

## 五、改进建议（按 ROI 排序）

1. **P2 修复（成本最低、收益最大）**：`chunker.py` 在合并 meta 时把 `page_index` 也并入 `slide_indices`（或新增 `page_indices`）。让 KP 能回溯到具体页。
2. **P3 修复**：`loaders.py:_make_title` 加一行启发式——若首行匹配 `^©\s*Copyright|CONFIDENTIAL` 则跳过取下一行。
3. **P1 运维**：给 Celery worker 加 `--autoreload`（开发环境）或在 README 增加「修改 ingestion 代码后必须重启 worker」。
4. **P5 OCR**：对扫描型/PPT 导出 PDF 增加 fallback——`pypdf` 抽出文本 < 阈值（如平均每页 < 100 字）时切到 `paddleocr` 或调用 vLM。当前 PDF 平均 264 字/页就在边界附近。
5. **P6 prompt 优化**：
   - 加一条规则：「如果文中以编号或并列结构出现 N 个项，请按完整 N 个分别抽取 KP」
   - 营销/介绍类文档用单独 prompt 模板，允许"产品功能"和"价值主张"类 KP，不强求"术语-定义"结构。
6. **dedup 阈值**：把 0.85 拆成名 + 定义两路相似度，或对"销售技巧"等近义稠密词域做白名单二次校验。

## 六、修复 P2/P3/P6 后的验证（doc_id=11，重跑结果）

修复点：
- **P2** `chunker.py`：合并 meta 时同时收集 `page_index`，写入 `page_indices` 与 `source_indices`
- **P3** `loaders.py:_make_title`：跳过 `©|Copyright|CONFIDENTIAL|All Right Reserved|纯页码` 的页脚行
- **P6** `extractor.py` prompt：新增「枚举完整性规则」——若文中出现 N 项列举（如"四种模式"、"5 阶段"），必须输出 N 个 KP

### 修复后新暴露的问题

| 问题 | 表现 |
|---|---|
| **枚举仍漏 1 项** | AI 四种模式只抽出 3 个（#133/134/135），**第 4 个「学习导师」仍漏**——因为 4 个并列项被 batch 切分跨了 LLM 调用窗口，单 batch 内 LLM 看不到完整 4 项 |
| **定制化三档只抽 2 档** | #129「低定制化」、#130「中定制化」抽到，**漏「高定制化」** |
| **主题卡 7 类只抽 5 类** | 抽到领导他人/沟通合作/效能提升/思维工具/销售类，漏「领导业务」「领导自我」；且未保留 "7 类 239 卡" 的数字明细 |
| **过细问题** | #184-188 (需求识别能力/痛点挖掘/业务理解/方案设计/谈判推进)，本来只是销售目标定义里的支撑能力词，被独立抽成 KP，颗粒度过细 |

### 进一步建议

1. **chunker 不要在并列结构中间切**：识别 `①…②…③…④` / `1./2./3.` / `高/中/低` / `第N` 等枚举标记，强制让一个完整枚举留在同一 chunk 内（避免跨 batch 漏项）
2. **数字保留规则**：prompt 加一句「定义中出现的数字（N 张、N 类、X%）必须原文保留」
3. **粒度上限**：prompt 加「同一段落内不超过 6 个 KP」防止过细

## 七、结论一句话

> 这份以视觉为主的市场介绍 PDF，**修复前**系统抽出 25 个 KP（覆盖 ~76% 核心概念），系统性漏掉学员端流程、四模式之一、SIMUGO 算法第 5 步等关键内容；**修复 P2/P3/P6 三处**后 candidate 涨到 89、new_kp 涨到 78，**冲突五阶段、SIMUGO 算法五步、销售三挑战等列举内容全部补齐**，但仍残留「跨 batch 枚举漏项」「数字明细丢失」「过细粒度」三类问题——这是下一轮 chunker + prompt 调优的目标。
